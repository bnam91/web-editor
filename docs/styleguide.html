디자인 토큰 스타일가이드

css/ 16개 파일에서 추출한 GODITOR 디자인 토큰 정본 문서. 토큰명을 클릭하면 클립보드에 복사됩니다. 사용횟수는 var(--token) 참조 수(css 전역)이며 2026-06-14 스냅샷입니다 — fable이 코드를 계속 정리 중이라 ±변동하며, 정확값은 scripts/check-design-tokens.sh로 재집계하세요. live 실가동 / bridge 별칭·연결 / dead 미사용·중복 / runtime JS가 런타임 주입 으로 구분했습니다.

⚠️ 알려진 이슈 · 디자인 부채

현재 코드 상태를 과장 없이 정리. 가이드를 보기 전에 먼저 알아둘 맥락입니다.

1. 색·치수 "정답표"가 두 개라서 한쪽은 거의 안 쓰임

💬
쉽게 말하면 — 누군가 색·간격을 깔끔하게 정리한 공식 정답표(A)를 만들어 뒀는데, 정작 화면을 그릴 땐 다른 사람이 따로 만든 정답표(B)를 씁니다. 둘이 적어둔 값은 사실 거의 똑같아요(예: 같은 파란색 #2d6fe8을 A·B가 각자 적어둠). 그래서 표 A는 만들어만 놓고 먼지 쌓인 상태고, 나중에 색을 바꾸려면 "어느 표를 고쳐야 먹히지?" 헷갈립니다. 정리하면 유지보수가 훨씬 편해집니다.
개발 디테일 보기

design-tokens.css는 2계층(primitive --p-* → semantic --color-*/--bg-*/--text-*)을 잘 정의해 뒀지만, 실제 앱은 editor-base.css의 평행 세트 --ui-*로 그려집니다.

  • --ui-accent-primary(#2d6fe8, 152회) ≈ --p-blue-600(#2d6fe8) 값 중복 — semantic --color-accent는 1회뿐.
  • --bg-*, --color-handle*, --color-danger* 등 semantic 다수가 각 1회(브리지용) → 사실상 dead.
  • --space-0~6, --radius-*(design-tokens) = 사용 0회, --ui-radius-*가 대체. 완전 dead.

2. 흐린 회색 글씨가 어두운 배경에서 잘 안 보이는 곳이 남아 있음

💬
쉽게 말하면 — 어두운 배경에 너무 연한 회색 글씨를 쓰면 눈이 침침해서 읽기 힘들죠. "정답표를 거친 글자색"은 이미 또렷하게 고쳐졌습니다(잘 보이게). 문제는 표를 안 거치고 코드에 직접 색을 박아 넣은 곳이 아직 많아서, 거기 회색 글씨들은 여전히 흐릿합니다. 눈 편하게 보려면 그 직접 박은 색들을 정답표 색으로 바꿔줘야 해요.
개발 디테일 보기

텍스트 토큰은 2026-06-11 패치로 AA 충족(--ui-text-muted #777→#9e9e9e 5.07:1, --ui-text-dim #555→#969696 4.59:1). 하지만 토큰 미경유 하드코딩 회색이 잔존:

  • 저대비 회색 직박이: #888 43회, #555 14·5회, #999 13회, #666 8회, #777 5회 등 — 다크 배경에서 WCAG AA 미달 다수.
  • css 전역 하드코딩 hex 총 926개 — 토큰화 이관 절반 이상 미완.
  • 의도적 비토큰(예외 — 고치지 말 것): .layer-item-type = raw #7e7e7e (editor-panels.css). 과거 var(--ui-text-dim)였으나 레이어 type 태그를 의도적으로 위계 다운(1단계 더 흐리게)하려 토큰 미경유 raw값으로 내림. --ui-text-dim(#969696)과 별개 톤이 목적이라 토큰화 대상 아님.

3. "같은 파란색"이어야 하는데 미묘하게 다른 파랑이 섞여 있음

💬
쉽게 말하면 — 버튼·선택·강조에 쓰는 대표 파란색은 딱 하나여야 통일감이 있는데, 실제론 비슷하지만 살짝 다른 파랑이 여러 개 섞여 있어요. 그래서 화면마다 파란색 톤이 미묘하게 달라 "왜 여긴 색이 좀 다르지?" 싶은 위화감이 생깁니다.
개발 디테일 보기

정본 --ui-accent-primary(#2d6fe8) 외 우회 블루 산재: #6b9eff 13회, #4f8ef7 8회, #1592fe 7회 등 → 활성 색 화면별 분열.

4. 이름만 부르고 실제 값은 안 정해둔 색이 있음 ✅ fable 해결됨

💬
쉽게 말하면 — "이 글자색은 기본글자색으로 해줘"라고 이름만 불러놓고, 정작 그 이름이 무슨 색인지는 어디에도 안 적어둔 경우예요. 지금은 비상용 기본값으로 대충 때워지지만, 상황이 바뀌면 색이 엉뚱하게 깨질 수 있는 불안한 부분입니다. (과거에 실제로 이렇게 깨졌다가 고친 적 있음.)
개발 디테일 보기

--ui-text-primary(2회), --ui-text-secondary(2회)는 :root 정의 없음 → var() fallback 의존. 과거 --ui-text-main 동일 사유로 깨져 별칭 수습 전례.

5. 그림자(입체감) 규칙이 따로 없어서 그때그때 손으로 넣음 ✅ fable 해결됨

💬
쉽게 말하면 — 팝업·모달이 떠 있을 때 주는 그림자(떠 보이는 입체감)에 정해진 규칙이 없어서, 만드는 곳마다 그림자를 따로따로 손으로 넣습니다. 그래서 어떤 창은 진하고 어떤 창은 연해 입체감이 들쭉날쭉해요. 그림자 규칙을 하나 정해두면 깔끔하게 통일됩니다.
개발 디테일 보기

--shadow-* / --elevation-* 토큰 전무. 드롭다운·모달·팝오버 그림자가 전부 ad-hoc box-shadow 리터럴 → depth 일관성 미보장.

6. 색상 입력칸의 hex 표기가 곳마다 다름 (#ffffff vs ffffff)

💬
쉽게 말하면 — 색을 입력하는 칸이 어떤 곳은 ffffff(# 없이 6자리)로, 어떤 곳은 #111111(# 붙여 7자리)로 표시돼요. 심지어 같은 패널 안에서도 배경색 칸과 글자색 칸이 서로 다르게 보입니다. 이유는 색 입력칸을 만든 방식이 두 가지(신·구 버전)가 섞여 있어서예요. 신형은 Figma처럼 # 없이 + 투명도(%)칸까지 있고, 구형은 # 붙은 옛날 방식. 보기에 들쭉날쭉하고, 사용자가 "#을 넣어야 하나 말아야 하나" 헷갈립니다. 전부 신형으로 맞추면 깔끔하게 통일돼요.
개발 디테일 보기

색 입력 UI 2세대 공존:

  • 신형 정석 헬퍼 colorFieldHTML() (color-picker.js) — 46곳 사용. # 없이 대문자 6자리(FFFFFF) + 알파(%) 입력. 입력 시 # 자동 제거·대문자화. = Figma 방식·정본.
  • 구형 인라인 — 최소 6곳(prop-text-template 2·prop-section 1·prop-simple-card 3). # 포함 7자리(#111111), maxlength="7", 알파 없음, 각 파일에 손으로 작성.
  • 증상: prop-section.js에서 배경색=신형(FFFFFF) / Heading색=구형(#111111) → 한 패널 내 표기 분열.

제안: 구형 6곳을 colorFieldHTML() + wireColorField()로 교체해 # 없는 6자리 + 알파로 일원화.

🔧 고칠 후보 — fable 일괄 요청 대기열

현빈이 실사용 중 발견한 구체적 UI 일관성·접근성 패치 항목 + 전방위 감사 발굴분. 위 이슈(시스템급 부채)와 달리 바로 손볼 수 있는 개별 건입니다. 각 행에서 상태(미정·대기·진행중·승인·반려)와 코멘트를 직접 지정할 수 있고(브라우저에 자동 저장), 📋 전체 복사로 마크다운을 떠서 고디터-fable에 한 번에 요청하면 됩니다. (이 문서는 앱 코드를 직접 고치지 않습니다 — fable 세션이 일괄 처리)

#위치증상제안 수정상태코멘트

더 발견하면 계속 추가됩니다. 상태·코멘트는 이 브라우저(localStorage)에 저장돼 새로고침해도 유지됩니다.

UI Shell

실제 앱 크롬(패널·보더·텍스트·버튼)을 굴리는 실가동 정본 세트. editor-base.css에 직접 정의. GODITOR의 다크 UI는 사실상 이 토큰들로 만들어집니다.

Accent · Selection

주요 액션 블루와 캔버스 블록 선택 색. --sel-*는 줌 무관 1px 아웃라인을 위해 --color-selection을 브리지합니다.

Status

위험/성공/저장 상태 색.

Category 팔레트 (A/B 베리에이션)

A/B 베리에이션 카테고리 뱃지 배경색 --ui-cat-a~e(+hover) — DS-16, 2026-06-14 신설. 비텍스트 뱃지 배경이라 대비 무관. A=파랑(accent-primary 동치)·B=주황·C=초록·D=보라·E=빨강. 각 1회 사용(claude-pm/editor-panels).

Primitive 팔레트

--p-* 원형 값. semantic 계층의 재료지만 대부분 1회(semantic이 1번 참조)만 쓰임. 회색 21단계가 촘촘히 난립(중복 톤 다수).

Neutral / Gray (21단)

Blue · Sky · Cornflower (selection/accent 원형)

Status · Content · Canvas 원형

Semantic 계층 (대부분 dead)

design-tokens.css가 의도한 "이상적" 의미 토큰. 좋은 설계지만 --ui-*에 밀려 각 1회 또는 0회만 사용. 향후 통합 시 살릴지/버릴지 결정 대상.

Preset · 캔버스 콘텐츠

사용자가 편집하는 블록 내부 디자인 기본값(밝은 배경 전제). UI 크롬과 분리 관리 — 토큰 강제 적용 금지 영역. JS가 inline으로 주입.

Runtime 토큰 (JS 주입)

:root에 없고 JS가 런타임에 요소별로 주입하는 동적 변수. 줌 보정·표 스타일·텍스트 이펙트·말풍선색 등. 값은 상황에 따라 변동. 또한 theme-system.js는 위 UI Shell --ui-* 토큰 전체를 런타임에 덮어쓸 수 있습니다(= 테마 전환 기능). 즉 정적 정본값은 "기본 테마"이고, 사용자가 바꾸면 JS가 그 위에 올립니다.

타이포그래피

✅ fable이 폰트 스케일을 토큰화 완료(2026-06-13, 내 요청 반영) — --ui-fs-9 ~ --ui-fs-28 9단. 더 이상 px 리터럴 역산이 아니라 토큰 기준입니다. 폰트는 2종: UI 크롬 = Pretendard 스택, 캔버스 콘텐츠 = Noto Sans KR(--preset-*-family). weight도 --ui-fw-medium/semibold/bold 토큰화됨.

UI 크롬 스케일 (--ui-fs-*)

캔버스 콘텐츠 (.tb-*, Noto Sans KR · 사용자 편집값)

weight 토큰 (--ui-fw-*)

line-height 토큰 (--ui-lh-*)

⚠️ 남은 부채: 18px는 토큰 없이 리터럴 5곳 난립(16/20 중 흡수 미정), --ui-fs-28은 정의됐으나 var 0회(dead). ✅ line-height 토큰 신설됨 (DS-22, 2026-06-14): --ui-lh-tight/normal/relaxed = 1.3/1.4/1.6 — 정의만, 치환은 2단계(현재 var 0회). 캔버스 콘텐츠(.tb-*/--preset-*)는 범위 밖. → 치환 대기열.

Radius

UI 크롬 모서리 스케일. 실가동은 --ui-radius-*(editor-base). --radius-*(design-tokens)는 동일 값이나 사용 0회.

Spacing / 간격

4px 기반 간격 스케일 --space-0~6이 design-tokens에 정의됐으나 직접 사용 0회 — css에 px 리터럴 3000+개가 미이관. 아래는 정의된 스케일(시각 바).

Shadow / Elevation

✅ fable이 그림자 토큰화 완료(이슈 #5 해소) — --ui-shadow-sm/md/lg/xl 4단, var 11회 사용 중. 아래는 실제 토큰 샘플(클릭 복사).

토큰 사용 규칙 (Do / Don't)

새 UI를 만들거나 색을 고칠 때 지켜야 할 실전 규칙. _context/DESIGN_SYSTEM.md · CLAUDE.md 기준. "하지 말 것"은 실제로 과거에 색이 깨지거나 어긋났던 사례들입니다.

✅ 이렇게 하세요

  • UI 크롬(패널·툴바·버튼)의 색·보더·텍스트는 실가동 --ui-* 토큰을 직접 쓴다.
  • 파란 강조/활성색은 오직 --ui-accent-primary(#2d6fe8) 하나로.
  • 본문 글자는 --ui-text(#e0e0e0) 이상의 또렷한 색. 흐린 색은 비활성·힌트에만.
  • 모서리·간격은 --ui-radius-* / --space-* 스케일 값에 맞춰.
  • hover·선택 같은 반투명 효과는 의미 토큰(--sel-color-fill 등) 우선.
  • 색을 새로 정의해야 하면 정해진 파일에만: 원형값→design-tokens.css, UI색→editor-base.css.

🚫 하지 마세요

  • 색을 코드에 직접(#888 같은 hex) 박지 말 것 — 토큰을 거칠 것. (지금 926개가 직박이 상태)
  • 어두운 배경에 --ui-text-dim(#555급) 본문 금지 — 대비 부족으로 안 보임.
  • 비슷한 다른 파랑(#6b9eff·#4f8ef7·#1592fe) 끌어다 쓰지 말 것.
  • 이름만 부르고 값을 안 정한 토큰(--ui-text-primary 등) 새로 만들지 말 것.
  • 선택 상태(selected)에 background 추가 금지 — outline만.
  • 텍스트 블록에 border-radius 추가 금지, box-shadow:0 0 0 2px식 선택 표시 금지.
  • 캔버스 콘텐츠(블록 내부 디자인)에 UI 토큰 강제 적용 금지 — 사용자 편집값임.

배경별 텍스트 대비 가이드

어떤 배경 위에 어떤 글자색을 올려야 잘 보이는지 짝지어 둔 표. 다크 셸(#252525) 기준 대비비를 함께 표기 — 본문은 4.5:1 이상이어야 또렷합니다(WCAG AA).

키보드 포커스 링 (a11y)

키보드 탐색 시 어디에 포커스가 있는지 보이게 하는 표준. :focus-visible 기준 — 마우스 클릭엔 안 뜨고 Tab 이동에만.

클립 컨테이너
inset -2px

✅ 규칙 (editor-base.css:798~)

  • button·prop-btn·align-btn·cv-chip·slider·[role=button/tab] = outline 2px solid --ui-accent-primary, offset 1px.
  • overflow:hidden 컨테이너 내부 항목은 offset -2px(inset)로 잘림 방지.

🚫 주의

  • outline:none 71곳 — 그중 비입력 포커서블(버튼·contenteditable)에 대체 표시 없으면 키보드 사용자 길 잃음.
  • input :focus를 border-color 전환만으로 처리하면 비텍스트 대비 3:1 미보장 → outline/box-shadow 병행.
